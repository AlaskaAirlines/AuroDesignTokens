## Breaking Change: `themeMetadata.iconUrl` Removed — `@aurodesignsystem/design-tokens` v9.0.0

### Summary

Version 9.0.0 removes the `iconUrl` field from `themeMetadata` in all distributed theme JSON files and adds two new analytics fields (`analyticsTrackingPrefix`, `analyticsBrandVariable`). Any consumer reading `themeMetadata.iconUrl` at runtime will receive `undefined` after upgrading.

---

### Why the change was made

Two new fields were required in `themeMetadata` to support Adobe Analytics theme identification at runtime:

| Field | Purpose |
|---|---|
| `analyticsTrackingPrefix` | Prefixes event names sent to Adobe Analytics (e.g. `AS_`, `HA_`, `Atmos_`) |
| `analyticsBrandVariable` | Identifies the active brand in analytics payloads (e.g. `AS-Light`, `HA-Light`) |

As part of this work, `iconUrl` was also removed. It was a development placeholder — a hardcoded Contentstack/CDN favicon URL that was never wired up to any UI or runtime logic. Confirmed unused by any consumer, it had no reason to remain in the distributed artifact.

---

### Why it is a breaking change

The `iconUrl` field was a **published, documented key** in the `themeMetadata` object of the distributed `dist/app/*.json` files. Its removal is a breaking change under semver because:

- Any code that reads `themeMetadata.iconUrl` will now receive `undefined` instead of a string URL.
- If that value is passed to an `<img src>`, a fetch, or any code expecting a non-nullish string, it will silently degrade or throw at runtime.
- The `iconurl` → `iconUrl` key mapping has also been removed from `figmaExportUtil.mjs`, meaning it will not reappear in future Figma-sourced builds even if re-added to token definitions.

---

### Affected files

| File | Change |
|---|---|
| `dist/app/alaskaLight.json` | `iconUrl` removed; `analyticsTrackingPrefix: "AS_"`, `analyticsBrandVariable: "AS-Light"` added |
| `dist/app/hawaiianLight.json` | `iconUrl` removed; `analyticsTrackingPrefix: "HA_"`, `analyticsBrandVariable: "HA-Light"` added |
| `dist/app/atmosLight.json` | `iconUrl` removed; `analyticsTrackingPrefix: "Atmos_"`, `analyticsBrandVariable: "Atmos-Light"` added |
| `dist/app/test.json` | `iconUrl` removed; `analyticsTrackingPrefix: "Test_"`, `analyticsBrandVariable: "TEST"` added |
| `scripts/figmaExportUtil.mjs` | `iconurl` key mapping removed; two new analytics key mappings added |
| `tokenDefinitions/manualTokens/app/*.json` | Source token files updated to match |

---

### Consumer impact

**Low risk overall.** `iconUrl` was confirmed unused by any known consumer at the time of removal. However:

- **If you read `themeMetadata` from the dist JSON** and access `.iconUrl`, your code will silently receive `undefined`. Add a check or remove the reference.
- **If you have TypeScript types derived from the dist schema**, regenerate them — `iconUrl` will no longer appear in the inferred type.
- **There is no migration path for `iconUrl`** — the value is gone and will not be reintroduced. If your application genuinely needs a theme favicon URL, source it from your own CDN or hardcode it per-theme in your application layer.

---

### Required action for consumers

1. Search your codebase for any access of `themeMetadata.iconUrl` (or `theme_metadata.icon_url` if you've snake_cased the keys).
2. Remove or replace those references before upgrading to `>=9.0.0`.
3. Optionally, take advantage of the new `analyticsTrackingPrefix` and `analyticsBrandVariable` fields if your app integrates with Adobe Analytics.
